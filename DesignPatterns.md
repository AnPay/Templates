| **Question**                                           | **Strong Answer (Concise + Impactful)**                                                                                                                                                                                                                                                                                                                        |
| ------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **What design patterns have you used?**                | I’ve used several design patterns like Strategy, Factory, Observer, and Singleton. One pattern I used extensively is the Strategy pattern in our inference pipeline to handle multiple backends and decoding approaches dynamically.                                                                                                                           |
| **Explain how you used Strategy pattern.**             | We had multiple variations in our inference pipeline—different hardware backends (NPU/GPU) and decoding strategies (standard vs speculative). We defined a common `InferenceStrategy` interface, and the Pipeline Controller acts as the context. Based on runtime configuration, we inject the appropriate strategy, allowing flexible and modular execution. |
| **Why Strategy over if-else?**                         | If we used if-else, the pipeline controller would become tightly coupled with all variations, leading to large conditional blocks and frequent modifications. Strategy decouples execution from decision-making, aligns with Open/Closed Principle, and makes the system extensible and easier to maintain.                                                    |
| **How do you select strategy at runtime?**             | Strategy selection is configuration-driven. Based on inputs like model size, hardware backend, and optimization flags, we use a factory-like component to map configuration to the appropriate strategy implementation. The controller only executes the selected strategy.                                                                                    |
| **What is the role of Pipeline Controller?**           | It acts as the context in the Strategy pattern. It does not contain decision logic—it simply executes the injected strategy, keeping orchestration separate from implementation details.                                                                                                                                                                       |
| **What are the trade-offs of Strategy pattern?**       | It increases code complexity due to multiple classes, introduces indirection which can make debugging harder, and requires careful configuration management. However, these are acceptable given the flexibility and scalability benefits.                                                                                                                     |
| **Did you combine Strategy with other patterns?**      | Yes, we combined Strategy with Factory pattern. Strategy handles execution logic, while Factory handles selection and instantiation based on configuration. This keeps responsibilities clean and improves extensibility.                                                                                                                                      |
| **Why not use templates (compile-time polymorphism)?** | Templates provide zero runtime overhead but require decisions at compile time. Our system needs runtime flexibility to switch strategies based on configuration. Also, templates can increase binary size due to multiple instantiations.                                                                                                                      |
| **Why not use plugin architecture?**                   | Plugin systems offer high flexibility and independent deployment, but they introduce complexity in deployment, versioning, and security. For our use case, Strategy + Factory provided sufficient flexibility with lower overhead.                                                                                                                             |
